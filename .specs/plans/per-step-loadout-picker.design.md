# Design: Per-Step Loadout Picker with Complexity-Aware Sizing and Pipeline-Owner Isolation

Generated by /office-hours on 2026-04-24
Branch: main
Repo: fernandoxavier02/skill-advisor
Status: DRAFT
Mode: Builder

## Problem Statement

Three issues hurt /advisor usability today:

1. **Fixed-size loadouts.** The router always emits a ~4-step pipeline (clarification → planning → implementation → verification) regardless of task scope. A simple one-liner fix gets the same heavy flow as a multi-slice refactor. Users either abandon the gate or blast through it, defeating the point of review.
2. **Cross-plugin mixing.** The router freely composes skills from plugins that have their own internal orchestration (superpowers, pipeline-orchestrator, kiro, sdd, compound-engineering). Pulling `ce-plan` into a loadout that ends in `/sdd:implement` breaks both plugins' contracts — each assumes it owns the full flow and carries context the other cannot consume. The gate completes with a technically valid JSON that nonetheless produces garbage when executed.
3. **Invisible pipelined plugins.** When the router does NOT mix owners, it also does not surface pipelined plugins at all — it composes standalone skills as if the orchestrated plugins did not exist. A user who would benefit from kiro's 3-phase flow for a new spec never sees the option unless they name the plugin explicitly. This caps the plugin's value because the router is not catalogued by functionality.

## What Makes This Right

A loadout proposal becomes a menu of calibrated choices instead of a monolithic prescription. Each step shows the router's pick alongside 2-3 alternatives with one-line explanations, selected via the native `AskUserQuestion` arrow-key menu we just unblocked. The user stays in control at every step without leaving the gate. Pipelined plugins are treated as atomic units — once chosen, they collapse the entire loadout to their native flow, so they execute in the environment they expect.

## Constraints

- Must reuse the `AskUserQuestion` bootstrap established by 4a820d3 (no new tool-loading contract)
- Must not break the existing `gate_output` JSON schema consumed by `commands/advisor.md` Step 7 pre-check
- Test suite stays at `node --test tests/*.test.js`; new regression tests follow the same pattern
- The change is additive to the index format: new field, no rename of existing fields
- CANONICAL_FLOWS seed list is small and hand-maintained; v1 supports 6 plugins

## Premises

1. Every skill in the index is either `standalone` or `pipeline_owner=<plugin-id>`. The classification comes from a curated list `PIPELINE_OWNERS` in `lib/constants.js`, not from heuristics or plugin metadata inference.
2. The router outputs an explicit `task_complexity` of `"simple" | "medium" | "complex"` in its JSON. Complexity informs default loadout size when composing standalone skills.
3. Default loadout size when composing standalone is a strict bound: simple 1-2 skills, medium exactly 3, complex 4-5. The router must pick an integer within the range; no "flexible" interpretation.
4. A loadout cannot mix skills from two different `pipeline_owner` values. Rejection is handled at two checkpoints: (a) router must self-validate before emitting; (b) gate revalidates on input and, if the router slipped, re-spawns the router once with `last_error: "mixed_pipeline_owners"`. If the second attempt is still invalid, emit `gate_output.error: "invalid_loadout"` with `decision: "cancel"`. This retry budget is **separate** from the malformed-JSON retry described in Feasibility Notes — each failure mode has its own single-retry allowance.
5. The gate collects decisions via one `AskUserQuestion` per position, offering the router's recommendation plus 2-3 alternatives, each with a one-line explanation.
6. Swap rules: standalone → standalone is a **local positional substitution**. The `depends_on` arrays stay position-indexed and are NOT rewritten; the swapped skill is assumed to satisfy the same role as the original. Selecting any pipeline-owned skill at any position collapses the entire loadout to that plugin's canonical flow in `CANONICAL_FLOWS`.

## Approaches Considered

### Approach A — Ship now (chosen)

Index tagging + per-step picker + collapse-on-pipeline-owned. No re-routing of downstream when a standalone is swapped locally.

- `lib/constants.js`: add `PIPELINE_OWNERS` array and `CANONICAL_FLOWS` object
- `lib/build-index.js`: tag each entry with `pipeline_owner: string | null`
- `agents/advisor-router.md`: emit `task_complexity` and per-entry `alternatives`; never mix owners; prefer standalone composition
- `agents/advisor-gate.md`: Moment 1 becomes a per-position loop of `AskUserQuestion`; collapse detector replaces loadout on pipeline-owned pick

### Approach B — Ship complete

A plus re-routing: gate calls router again with a `locked_prefix` when a standalone is swapped mid-loadout. Pipeline always coherent end-to-end. Rejected for v1 on effort/risk grounds — B is additive and can follow once A ships.

### Approach C — Inline edit via HTML artifact

Interactive comparison board in a browser tab. Rejected as it breaks the AskUserQuestion pattern we just fixed and adds a browse dependency.

## Recommended Approach

**Approach A.** Covers 90% of the value at roughly half the effort of B. The swap-downstream-incoherence edge case is bounded: most useful swaps either fall into the pipeline-owned collapse path (which A handles cleanly) or into substitutions where downstream does not actually depend on the prior step's specific identity. If a week of real use surfaces frequent coherence breakage, B is a clean additive upgrade.

### Pipeline-owner functional fingerprint (triage-first recognition)

For the router to recognize early that a task maps to a pipelined plugin end-to-end (instead of trying to compose standalone and failing into a collapse later), each owner carries a **functional fingerprint**: a structured description of what kind of work the plugin handles, stored in `lib/constants.js` alongside PIPELINE_OWNERS and CANONICAL_FLOWS.

```js
const PIPELINE_FINGERPRINTS = {
  'superpowers': {
    best_for: 'Multi-step feature development with brainstorm-plan-execute-verify discipline.',
    typical_tasks: ['design new feature', 'refactor with planning', 'systematic implementation'],
    not_for: ['one-line fixes', 'quick lookups', 'simple bug fixes'],
    complexity_match: ['medium', 'complex']
  },
  'pipeline-orchestrator': {
    best_for: 'Task with formal classification, gates, and adversarial review (bug fix / feature / audit / security).',
    typical_tasks: ['bug fix with root cause analysis', 'feature with review batches', 'code audit'],
    not_for: ['exploratory work', 'plain implementation without quality gates'],
    complexity_match: ['medium', 'complex']
  },
  'kiro': {
    best_for: 'Kiro-style Spec-Driven Development — requirements → design → tasks → impl with 3-phase approvals.',
    typical_tasks: ['new spec', 'validate implementation gap', 'structured feature dev'],
    not_for: ['ad-hoc fixes', 'projects without .kiro/ scaffolding'],
    complexity_match: ['complex']
  },
  'sdd': {
    best_for: 'Spec-Driven Development with LLM-as-Judge verification.',
    typical_tasks: ['brainstorm → plan → implement with automated quality checks'],
    not_for: ['simple edits', 'work that does not benefit from judge verification'],
    complexity_match: ['medium', 'complex']
  },
  'compound-engineering': {
    best_for: 'Full compound workflow (brainstorm → plan → work → commit → PR).',
    typical_tasks: ['feature with structured delivery', 'end-to-end development loop'],
    not_for: ['diagnostic-only tasks', 'refactors without commit scope'],
    complexity_match: ['medium', 'complex']
  }
};
```

**Router contract addition.** Before composing any loadout, the router scans PIPELINE_FINGERPRINTS and checks whether the task description matches any owner's `best_for` + `typical_tasks` signatures AND the declared `task_complexity` ∈ `complexity_match`. If a match scores above a confidence threshold (e.g., ≥ 0.7 semantic similarity), the router SHOULD emit the canonical flow for that owner directly as the primary recommendation, with the alternatives array containing other pipelined plugins' canonical flows plus a "standalone composition" synthetic alternative. If no fingerprint matches, the router proceeds with standalone composition under the complexity size bounds.

**Why this matters.** Without the fingerprint, the router's only path to a pipelined plugin is via the gate's collapse-on-pick behavior — which requires the user to already know the plugin exists and ask for it. The fingerprint flips this: the router can recommend `/kiro-spec-quick` end-to-end from a description like "create a spec for the auth refactor," without the user having to name kiro.

**Tests (listed together with T1-T8 in Testing Strategy below):**
- **T9:** Parity `Object.keys(PIPELINE_FINGERPRINTS) === PIPELINE_OWNERS`.
- **T10:** Router fixture outputs for 3 task prompts (mapping to kiro / superpowers / pipeline-orchestrator) carry `matched_fingerprint: "<owner>"` in the JSON and `loadout` deep-equals the materialized canonical flow of that owner.

**Fingerprint matching mechanism (clarified).** The router is a single Claude LLM call, not an embedding system. "Match" is a prompt-level judgment, not a similarity score: the router prompt includes the full PIPELINE_FINGERPRINTS object and a rule — "if the user task clearly matches any owner's `best_for` and `typical_tasks` AND the declared complexity is in that owner's `complexity_match` array, emit the canonical flow as primary and set `matched_fingerprint: <owner>`. Otherwise set `matched_fingerprint: null` and compose standalone." The router's judgment is the threshold; tests validate the output, not the reasoning.

**Router output schema addition:** `matched_fingerprint: string | null` at the top level. Enables T10 assertion and gate-level logging. Backward-compat: consumers treat missing field as `null`.

## Behavior Contracts

### Router composition rules

1. Classify `task_complexity` explicitly. Sizing targets: simple 1-2 skills, medium 3, complex 4-5. Sizing applies only when composing `pipeline_owner: null` skills.
2. Prefer standalone skills when composing. A pipeline-owned skill is only recommended when one of the following holds:
   - The task is complex AND no viable standalone composition covers the flow
   - The user's task description explicitly names the plugin ("use kiro for this", "via superpowers")
3. A loadout must contain either all standalone skills, or all skills from the same `pipeline_owner`. Mixed loadouts are invalid and must be rewritten before emission.
4. Every loadout entry must carry an `alternatives` array of 2-3 options. Alternatives respect the same owner rule: in a standalone loadout, alternatives are all standalone; in a pipeline-owned loadout, alternatives propose alternate canonical flows (e.g., `sdd:plan` vs `superpowers:writing-plans` — each of which would trigger its own collapse).

### Gate per-step picker

**Initial state.** The gate inspects the incoming router loadout before starting the per-step loop:
- If every entry has `pipeline_owner === null` (pure standalone loadout), run the per-step loop normally.
- If every entry shares the same non-null `pipeline_owner` (router already emitted a canonical flow), **skip the per-step loop** and present a single summary menu (Sim/Nao/Alterar/Sugerir) over the whole flow. Rationale: there is nothing to swap per-step in a canonical flow.
- If the loadout mixes owners (invalid), trigger the router re-spawn described in premise 4.

**Per-step loop (standalone loadout only).**

1. After the existing tool-loading bootstrap and standard loadout summary box, iterate `N = 1..loadout.length`.
2. Each iteration runs `AskUserQuestion` with:
   - Question: `"Step ${N}: qual skill executa ${role}?"`
   - Options: recommendation (labeled "Recomendado") + up to 3 alternatives. Each option carries the `one_line` as the `description` field. A final option `Voltar` is appended (always at least 2 options after Voltar, per existing gate rule).
3. If the user picks the recommendation or a same-owner alternative (both null, or both equal to current owner), record the choice at position N, continue to N+1.
4. If the user picks a cross-owner alternative (`pipeline_owner !== current owner`), **collapse immediately**: replace the loadout with the materialized canonical flow (see "Canonical flow materialization" below), exit the per-step loop, and present a single summary menu (Sim/Nao). A Sim answer emits `gate_output` with `decision: "custom"` and the collapsed loadout; Nao emits `decision: "cancel"`.
5. `Voltar` semantics: aborts the per-step loop unconditionally, discards all per-step picks accumulated so far, restores the router's original loadout, and re-presents the classic full-loadout menu (Sim/Nao/Alterar/Sugerir). It does NOT undo picks one at a time; it is a single-step escape.
6. End-of-loop emission: `decision: "custom"` if at least one alternative was picked during the loop, `decision: "approve"` if every position kept the recommendation. In both cases, `loadout` reflects the final positions.

**Canonical flow materialization.** When collapsing, the gate must produce full loadout entries (router consumers expect `role`, `category`, `confidence`, `reason`, `depends_on`). Defaults:
- `role` = category value of the skill in the index (looked up via the Step-1 index snapshot)
- `category` = index category
- `confidence` = `1.0`
- `reason` = `"Parte do fluxo canônico de ${owner}."`
- `depends_on` = `[N-1]` for positions N > 1, `[]` for position 1

**Decision enum (SSOT).** `gate_output.decision` ∈ `{approve, cancel, custom, alternative}`. This design adds no new values: `custom` covers both per-step picks and collapse; `alternative` stays reserved for the Alterar flow of the original gate (unchanged). The Decision Routing Contract table in `commands/advisor.md` Step 6 is unaffected.

### Backward compatibility

Consumers reading `gate_output` or index entries that were built before this change must keep working. Migration rules:
- Missing `pipeline_owner` on an index entry → treat as `null` (standalone).
- Missing `alternatives` array on a router loadout entry → gate skips the per-step picker and uses the classic single-menu gate (legacy path). This means a router still emitting the old schema produces the old UX, no crash.
- Missing `task_complexity` on router output → gate proceeds without complexity-aware hints; no per-step bounds enforcement.

### Moment 2 (spec generation) stays unchanged. The spec skill choice is still a single menu.

## Correctness Properties

- **CP-1:** No emitted loadout contains skills from two distinct non-null `pipeline_owner` values. (Validated by router output check + gate post-collapse check.)
- **CP-2:** Loadout size respects `task_complexity` bounds for standalone-only loadouts. (Router emits violating sizes → test flags.)
- **CP-3:** Every alternative in the gate menu maps to an installed skill in the Step-1 index snapshot (no phantom entries).
- **CP-4:** Collapsing into a canonical flow produces a loadout of length `CANONICAL_FLOWS[owner].length` with every `invocation` matching the mapped array.

- **T3:** Router never emits a loadout with entries from two different non-null `pipeline_owner` values (fixture outputs violating this must fail schema validation).
- **T4:** Gate collapse: given a fixture standalone loadout and a simulated cross-owner pick, the post-collapse `gate_output.loadout` deep-equals the materialized canonical flow (full entries with role/category/confidence/reason/depends_on populated per the default rule).
- **T5:** Gate per-step pick with same-owner (or both-null) alternative: only position N is patched; all other positions' invocation and depends_on remain byte-identical.

- **T7:** Parity invariant: `Object.keys(CANONICAL_FLOWS).sort()` deep-equals `PIPELINE_OWNERS.slice().sort()`. Catches list drift.
- **T8:** Backward compat: router output missing `alternatives` → gate falls back to classic single-menu flow (no per-step picker); router output missing `task_complexity` → no size validation applied; index entry missing `pipeline_owner` → treated as null; router output missing `matched_fingerprint` → treated as null (no canonical-flow auto-promotion). All paths emit valid `gate_output`.
- **T9:** Parity: `Object.keys(PIPELINE_FINGERPRINTS).sort()` deep-equals `PIPELINE_OWNERS.slice().sort()`.
- **T10:** Fingerprint-match routing: for 3 fixture task prompts mapping to kiro / superpowers / pipeline-orchestrator, the router output carries `matched_fingerprint: "<owner>"` AND `loadout` deep-equals `CANONICAL_FLOWS[owner]`.

## Implementation Feasibility Notes

- **AskUserQuestion option limits.** The tool accepts 2-4 options with `label` ≤ 12 chars. The per-step picker with 1 recommendation + 3 alternatives + Voltar = 5 options exceeds the limit. **Mitigation:** cap alternatives at 2 when Voltar is included (4 total), or omit Voltar and rely on the user selecting Cancel/Nao at the end summary. Chosen: cap alternatives at 2 per step (plus recommendation plus Voltar = 4). Router still emits up to 3 for telemetry headroom; gate picks the top-confidence 2 for display.
- **AskUserQuestion has no native "back" primitive.** Voltar is a regular option labeled `Voltar` that the gate detects by label equality; there is no platform-level back gesture. Implementation must map the user's string return back to the enum value. Documented here so the implementer does not look for a missing API.
- **Router prompt size / reliability.** The new fields (`task_complexity`, `matched_fingerprint`, per-entry `alternatives` with 2-3 options including `one_line`, `pipeline_owner` mirror) add roughly 200-400 tokens per loadout. On a 4-position loadout, worst case +1.6 KB of structured JSON. No evidence this breaks Sonnet's JSON reliability, but the gate adds a schema-validation step on router output with a single retry on malformed JSON before cancelling (separate from the mixed-owners retry in premise 4).
- **Alternatives ranking and display.** Alternatives in the router output are ORDERED: the most relevant comes first. The gate displays the first 2 (plus recommendation plus Voltar = 4 total, within AskUserQuestion's limit). No `confidence` field on alternatives — array order IS the ranking. Router prompt instructs "list alternatives from most to least aligned with the role."
- **Output language (`one_line` and user-facing strings).** Router emits `one_line` in Portuguese (pt-BR) to match this plugin's current user-facing convention (see `agents/advisor-gate.md` Rule 8: "Present in PT-BR for user-facing text, EN for JSON keys"). Tests assert on stable substrings, not full localization, to avoid locale-coupled fixtures.
- **Plugin cache refresh.** Version bump `0.3.1 → 0.3.2` alone does not force re-fetch for users. Document in the release notes: users run `/plugin update skill-advisor` or manually replace the cache directory. CI tests rely on source, not cache, so pipeline stays green during rollout.
- **No-viable-composition fallback.** If the router cannot build a compliant loadout (e.g., task needs capabilities only present in pipeline-owned plugins but the user's context asks for small/simple sizing), the router MUST emit a 1-position loadout recommending a single pipeline-owned entry with `task_complexity: "complex"` regardless of user framing, AND include a `reasoning` line explaining the upgrade. Router never returns an empty loadout.
