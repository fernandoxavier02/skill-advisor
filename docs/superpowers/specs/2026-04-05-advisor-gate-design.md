# Advisor Gate — Enforcement Agent for User Approval

**Status:** APPROVED (revised after adversarial review)
**Date:** 2026-04-05
**Author:** Fernando Xavier + Claude (brainstorm session)
**Revision:** 2 — incorporated 13 adversarial findings (3 CRITICAL, 4 HIGH, 4 MEDIUM, 2 LOW)

---

## Summary

Introduce `agents/advisor-gate.md` as a mandatory enforcement agent that sits between the advisor-router's loadout and any execution. No skill is invoked without passing through the gate. The gate uses a **recursive 4-option pattern** (Sim/Nao/Alterar/Sugerir) at two decision moments: loadout approval and spec generation tool selection.

---

## Problem

The current `/advisor` command presents a dry-run and offers 4 options (Execute/Modify/List/Cancel). This lacks:

1. **No alternative generation** — user can only tweak the single loadout, not see different approaches
2. **No brainstorming path** — user can't propose custom skills and discuss execution
3. **No spec generation** — execution happens without a documented plan with exact invocation moments
4. **No enforcement** — the command can theoretically skip approval

---

## Solution

### New Agent: `agents/advisor-gate.md`

- **Model:** Sonnet
- **Role:** Enforcement gate — blocks execution until user explicitly approves
- **Pattern:** Recursive 4-option approval (Sim/Nao/Alterar/Sugerir)

---

## Iteration Limits (CRITICAL — prevents infinite loops)

| Interaction | Max Rounds | After Limit |
|-------------|-----------|-------------|
| **Alterar** (Moment 1) | 3 | Only Sim/Nao available |
| **Sugerir** (Moment 1) | 2 | Only Sim/Nao available |
| **Alterar** (Moment 2) | 2 | Only Sim/Nao available |
| **Sugerir** (Moment 2) | 1 | Only Sim/Nao available |
| **Total gate interaction** | 15K tokens | Force Sim/Nao decision |

The gate tracks iteration counts internally and removes exhausted options from the menu.

---

## Enforcement Mechanism (CRITICAL)

The gate is enforced by **architecture, not instruction**:

1. `advisor.md` Step 7 (execution) ONLY runs if it receives a valid gate output contract with `decision != "cancel"` AND `gate_token` (a unique ID generated by the gate).
2. The execution logic checks: `if (!gate_output || !gate_output.gate_token) → STOP with error "Gate not invoked"`.
3. This is not a hook — it is a contract: the advisor.md code path physically requires the gate output to proceed.

If enforcement via contract proves insufficient in practice, escalate to a `PreToolUse` hook in a future iteration.

## Two Decision Moments

### Moment 1: Loadout Approval

After the advisor-router returns a loadout and the dry-run is presented:

```
┌─────────────────────────────────────────────┐
│  ADVISOR LOADOUT                             │
│                                              │
│  1. /investigate  [debugging]    ~5min       │
│     → Diagnose root cause                   │
│                                              │
│  2. /fix          [impl]        ~10min       │
│     → Apply targeted fix                    │
│     depends on: #1                          │
│                                              │
│  Estimated: ~15min | ~6K tokens              │
├─────────────────────────────────────────────┤
│                                              │
│  1) Sim      — executar este pipeline        │
│  2) Nao      — cancelar                      │
│  3) Alterar  — ver 3 alternativas  [N/3]     │
│  4) Sugerir  — montar pipeline customizado   │
│                                              │
└─────────────────────────────────────────────┘
```

`[N/3]` shows remaining Alterar rounds. When exhausted, option 3 is removed.

**Option behaviors:**

| Option | Action |
|--------|--------|
| **1. Sim** | Proceed to Moment 2 (spec generation) |
| **2. Nao** | Return `decision: "cancel"`. Pipeline stops. |
| **3. Alterar** | Re-spawn advisor-router requesting 3 alternative loadouts. Present alternatives numbered 1-3 with recommendation. User picks 1/2/3 or "0" to return to ORIGINAL loadout with 4-option menu (alternatives discarded). Each re-run generates FRESH alternatives. |
| **4. Sugerir** | Invoke brainstorming skill (see Brainstorming section). Result is converted to structured loadout via advisor-router. |

### "nenhuma" (option 0) after Alterar

When alternatives are presented and user picks "0" (nenhuma):
- Alternatives are **discarded**
- **Original** loadout is restored
- 4-option menu re-appears (with Alterar count decremented)
- User can Alterar again (fresh alternatives) or choose another option

### Moment 2: Spec Generation Tool Selection

After loadout is approved (via option 1, 3, or 4), the gate recommends a skill to generate the execution spec:

```
┌─────────────────────────────────────────────┐
│  PIPELINE APROVADO — GERACAO DA SPEC         │
│                                              │
│  Para documentar a execucao, recomendo:      │
│                                              │
│  /sdd:plan (context-engineering-kit)         │
│  Motivo: loadout com 4 skills cross-plugin,  │
│  decomposicao completa com steps e gates     │
│                                              │
├─────────────────────────────────────────────┤
│                                              │
│  1) Sim      — gerar spec com /sdd:plan      │
│  2) Nao      — executar sem spec (legacy)    │
│  3) Alterar  — ver alternativas  [N/2]       │
│  4) Sugerir  — escolher outra skill          │
│                                              │
└─────────────────────────────────────────────┘
```

**Option 2 (Nao) warning:** If user skips spec, display:

```
⚠️  Sem spec, a execucao sera no modo legacy (v1.0):
    - Sem documento de pipeline
    - Sem invocacoes exatas documentadas
    - Sem monitoramento por agentes futuros (v2.0)
    Confirma? (sim/nao)
```

**Recommendation logic:**

| Loadout Complexity | Recommended Skill | Reason |
|-------------------|-------------------|--------|
| Simple (1-2 skills) | `/superpowers:writing-plans` | Lightweight, fast |
| Medium (3-4 skills) | `/sdd:plan` or `/superpowers:writing-plans` | Balanced |
| Complex (5+ skills, cross-plugin) | `/sdd:plan` | Full decomposition |

**Handling variable planning skill availability:**

| Installed Count | Behavior |
|----------------|----------|
| 0 | Alterar disabled. Gate generates spec inline (fallback). Warning displayed. |
| 1 | Alterar shows the 1 skill with note "unica opcao disponivel" |
| 2 | Alterar shows 2 skills |
| 3+ | Alterar shows top 3 skills |

The gate queries the skill index for entries with `category: "planning"` or matching names (`*plan*`, `*writing-plans*`, `*spec*`).

## Brainstorming ("Sugerir") Specification

### Skill Selection

The gate checks for installed brainstorming skills in order:

1. `sdd:brainstorm` (context-engineering-kit)
2. `superpowers:brainstorming` (superpowers)
3. Any skill matching `*brainstorm*` in the index

**Fallback:** If no brainstorming skill is installed, the gate conducts inline brainstorming — asks the user one question at a time about which skills to include and in what order. Less structured than a dedicated skill but functional.

### Context Passed to Brainstorming

```
"O usuario quer montar um pipeline customizado para: {task}.
Ele quer escolher quais skills incluir e discutir a ordem
de execucao.

Skills disponiveis (top 20 do graph search):
{list from graph-search, including name, plugin, category, description}

Ajude-o a montar o pipeline ideal."
```

The "top 20 skills" comes from calling `graphSearch(tokens, graph, 20)` in `lib/graph-search.js` with the task description tokenized.

### Brainstorming Output → Structured Loadout

Brainstorming produces unstructured text. The gate converts it to a structured loadout by:

1. Extracting skill names mentioned in the brainstorming conclusion
2. **Validating each against the index** — skills not in the index are flagged and removed with warning
3. Re-invoking `advisor-router` with: "Convert this brainstorming result into a structured loadout: {brainstorm_summary}. Only include skills that exist in the index."
4. The router returns a standard loadout JSON

This keeps the router as the single authority for loadout structure.

## Pipeline Spec Requirements

The spec format is defined in the v2.0 design spec (`.specs/plans/skill-advisor-v2-orchestration-platform.design.md`, Section 7). The gate references that document as SSOT rather than duplicating it.

**Mandatory fields per phase (from v2.0 Section 7):**

| Field | Description |
|-------|-------------|
| Skill | Full skill name |
| Plugin | Source plugin |
| Invocation | Exact call: `Skill("name")` |
| Moment | When to run: `INICIO`, `APOS Fase N PASS` |
| Prompt | Complete prompt with `{fase_N.campo}` references |
| Input | What this phase receives |
| Output esperado | What this phase must produce |
| Gate de saida | Criteria to advance |

The gate passes a reference to this schema (not inline requirements) to the planning skill.

```markdown
The gate's decision is FINAL. Step 7 requires gate_token from gate output.
```

### Step 7 (updated)

```markdown
### Step 7: Handle gate decision

PREREQUISITE: gate_output must exist and contain gate_token.
If gate_output is missing or gate_token is absent → STOP with error.

Based on the gate's response:

- decision: "cancel" → log telemetry, STOP
- decision: "approve" | "alternative" | "custom":
  - If spec_path exists → read spec, execute per documented invocations
  - If spec_path is null (legacy mode) → execute loadout directly (v1.0 behavior)
```

## Gate Output Contract

```json
{
  "gate_token": "gate-<uuid>",
  "decision": "approve | cancel | alternative | custom",
  "moment2_decision": "approve | skip | alternative | custom",
  "loadout": [],
  "original_loadout": [],
  "spec_path": ".specs/pipelines/fix-login-2026-04-05.md | null",
  "planning_skill_used": "/sdd:plan | null",
  "brainstorm_summary": "... | null",
  "iterations": {
    "moment1_alterar": 0,
    "moment1_sugerir": 0,
    "moment2_alterar": 0,
    "moment2_sugerir": 0
  },
  "error": null
}
```

**Field explanations:**

| Field | Purpose |
|-------|---------|
| `gate_token` | Unique ID proving gate was invoked. Advisor.md checks this. |
| `decision` | Moment 1 outcome |
| `moment2_decision` | Moment 2 outcome. `"skip"` = user chose Nao (legacy). |
| `loadout` | Final approved loadout (may differ from original) |
| `original_loadout` | Router's original loadout (preserved for telemetry) |
| `spec_path` | Path to generated spec, or null if skipped |
| `planning_skill_used` | Which skill generated the spec, or null |
| `brainstorm_summary` | If Sugerir was used, summary of discussion |
| `iterations` | Count of each option used (for telemetry/limits) |
| `error` | Error message if router/brainstorm/planning skill failed, null otherwise |

**Error states:**

| Situation | error value | decision |
|-----------|-------------|----------|
| Router fails on Alterar re-run | `"router_failed: <message>"` | Loadout reverts to original, menu re-appears |
| Brainstorming skill not found | `"brainstorm_skill_not_found"` | Falls back to inline brainstorming |
| Planning skill crashes | `"planning_failed: <message>"` | Returns with `spec_path: null`, warning to user |
| Brainstorming produces invalid skills | `"invalid_skills_removed: [list]"` | Skills removed, warning shown, valid skills kept |

## v2.0 Agent Flow Positioning

In the current v0.1 architecture:

```
router → gate (Moment 1 + 2) → execution
```

In the v2.0 architecture (6 agents), the gate sits between clarifier and planner:

```
router → clarifier → gate (Moment 1) → planner (replaces Moment 2) → executor
```

When the `advisor-clarifier` agent is built (Phase 3, task 3.2), the gate's role narrows: it handles ONLY loadout approval (Moment 1). Spec generation (Moment 2) migrates to the planner agent. The gate does NOT handle clarification — that is the clarifier's job. By the time the gate sees the loadout, ambiguities are already resolved.

This is documented here so the gate is built with clean boundaries from the start.

## The 4-Option Pattern (Reusable)

This pattern becomes the standard approval mechanism for the advisor plugin:

```
┌─────────────────────────────────┐
│  [Content to approve]            │
├─────────────────────────────────┤
│  1) Sim      — aceitar           │
│  2) Nao      — rejeitar          │
│  3) Alterar  — ver opcoes [N/M]  │
│  4) Sugerir  — brainstorming     │
└─────────────────────────────────┘
```

**Display language:** PT-BR for user-facing labels (Sim/Nao/Alterar/Sugerir). EN for JSON keys and enum values (approve/cancel/alternative/custom). This mapping is fixed:

| User sees (PT-BR) | JSON value (EN) |
|-------------------|-----------------|
| Sim | `approve` |
| Nao | `cancel` |
| Alterar | `alternative` |
| Sugerir | `custom` |

## Constraints

- The gate is MANDATORY — enforced by gate_token contract, not just instruction
- No skill execution without a valid gate output
- Iteration limits: max 3 Alterar (Moment 1), max 2 Sugerir (Moment 1), max 2 Alterar (Moment 2)
- Box-drawing Unicode for visual consistency
- Gate does not invent skills — queries the index, validates against it
- Brainstorming uses existing installed skills; inline fallback if none available
- Brainstorming output validated against index before becoming loadout
- Spec format references v2.0 design spec as SSOT (no duplication)
- v2.0 migration path documented: gate narrows to Moment 1 only when planner agent arrives
